Claude Code — Bug Fix Report

Claude Code desktop app stopped responding after update 1.1.7714. Two separate root causes identified and resolved across Mac and Windows.

Date: 2026-03-23 Affected: Mac mini + Windows 11 Status: ⚠ Workaround Active

📋 Summary

After updating the Claude desktop app to version 1.1.7714, Claude Code stopped working entirely on both machines. The app would connect but Claude Code would never respond or load.

Investigation revealed two completely separate bugs — one in how the app deployed its remote agent binary to the Proxmox server, and one in a missing Windows feature required for local VM execution.

Bug 1 — Mac & SSH
Broken ccd-cli binary deployed to Proxmox passing forbidden flags when running as root
Bug 2 — Windows only
Hyper-V and Windows Hypervisor Platform not enabled — app couldn't spin up its local VM

How Claude Code desktop app works (SSH mode)

When you open a remote project via SSH, the desktop app does the following:

  1. SSHs into the remote server (root@<proxmox-server-ip>)
  2. Deploys its own agent binary to /root/.claude/remote/ccd-cli on the server
  3. Executes ccd-cli to start a Claude Code session in the remote working directory
  4. Communicates back to the desktop app over an RPC socket

This means if ccd-cli is broken or misconfigured, the entire session silently fails.

🔴 Symptoms Observed

Mac (SSH → Proxmox)

  • Claude Code appeared to connect but never became interactive
  • No error shown in the UI — just a stuck/loading state
  • Claude Code CLI worked fine when run directly from terminal on Proxmox
  • Logs on Proxmox (/root/.claude/remote/remote-server.log) showed the session starting but immediately dying

Windows

  • Hard error in the desktop app: HCS operation failed: HRESULT 0x80070005 (Access Denied)
  • App could not launch the local Hyper-V virtual machine it uses to sandbox Claude Code
  • Happened immediately on attempt to open any Claude Code session

🧬 Root Causes

Root Cause 1 — Broken ccd-cli binary (Mac + SSH flow)

Update 1.1.7714 bundled a broken version of ccd-cli (v2.1.78). When the desktop app deploys this binary to the Proxmox server and executes it, it passes the flag:

--allow-dangerously-skip-permissions

Claude Code explicitly refuses to start with this flag when running as root — it's a security guardrail. Since the Proxmox server runs everything as root, Claude silently exits on every launch attempt.

Why it was silent The desktop app doesn't surface the Claude process exit reason in the UI. It just shows a hanging/stuck state, making this very hard to diagnose without inspecting the remote server logs and binary directly.

Root Cause 2 — Hyper-V not enabled (Windows only)

On Windows, the Claude desktop app uses Hyper-V to run Claude Code inside a lightweight local virtual machine for sandboxing. The Windows Hypervisor features were not enabled on this machine, causing the Host Compute Service (HCS) to return Access Denied immediately.

Error message HCS operation failed: HRESULT 0x80070005 (Access Denied)

🔧 Fix 1 — ccd-cli Wrapper Script Applied

The broken binary at /root/.claude/remote/ccd-cli was replaced with a bash wrapper script. The wrapper:

  • Sets the correct Node.js PATH (via nvm)
  • Iterates over all arguments and silently drops the two bad flags
  • Passes all remaining arguments to the real Claude CLI script
bash#!/bin/bash
export PATH=/root/.nvm/versions/node/v24.14.0/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
ARGS=()
for arg in "$@"; do
    [[ "$arg" == "--dangerously-skip-permissions" ]] && continue
    [[ "$arg" == "--allow-dangerously-skip-permissions" ]] && continue
    ARGS+=("$arg")
done
exec /root/.nvm/versions/node/v24.14.0/bin/node /root/.claude/remote/ccd-cli.js "${ARGS[@]}"

🪟 Fix 2 — Enable Hyper-V on Windows Applied

Ran the following commands in PowerShell as Administrator, then rebooted:

powershellEnable-WindowsOptionalFeature -Online -FeatureName Microsoft-Hyper-V-All -NoRestart
Enable-WindowsOptionalFeature -Online -FeatureName HypervisorPlatform -NoRestart

Both commands returned RestartNeeded: True. After reboot, Claude Code on Windows worked immediately.

Permanence This fix is permanent. Hyper-V remains enabled across all reboots. No maintenance required.

🔒 Fix 3 — Immutable Lock on ccd-cli Applied

The desktop app can auto-redeploy ccd-cli when it reconnects or detects a version change, which would silently overwrite the wrapper and break everything again.

To prevent this, the wrapper was locked at the filesystem level using the Linux immutable attribute:

bash# Applied on Proxmox server
chattr +i /root/.claude/remote/ccd-cli

# Verify — look for 'i' flag in the output
lsattr /root/.claude/remote/ccd-cli
# ----i---------e------- /root/.claude/remote/ccd-cli

With the i flag set, no process — not even root — can modify, overwrite, rename, or delete this file until the flag is explicitly removed.

Important — before future Claude updates If Anthropic releases a patched version of ccd-cli that no longer passes the bad flags, you must remove the immutable lock before the app can update the file: 
ssh root@<proxmox-server-ip> "chattr -i /root/.claude/remote/ccd-cli"
Then reconnect the desktop app to let it redeploy. Verify the new binary doesn't pass the bad flags before removing the wrapper.

🚨 Recovery Guide

If Claude Code stops working again in the desktop app, follow these steps in order.

Scenario A — ccd-cli got overwritten (most likely)

Symptoms: worked before, now broken again after app update or reconnect.

1
Check if the immutable flag is still set 
ssh root@<proxmox-server-ip> "lsattr /root/.claude/remote/ccd-cli"

You should see ----i---------e-------. If the i is missing, the flag was removed.

2
Restore wrapper from backup 
ssh root@<proxmox-server-ip> "cp /root/.claude/remote/ccd-cli.working-backup /root/.claude/remote/ccd-cli && chmod +x /root/.claude/remote/ccd-cli && chattr +i /root/.claude/remote/ccd-cli"
3
Verify the file looks correct 
ssh root@<proxmox-server-ip> "cat /root/.claude/remote/ccd-cli"

Must start with #!/bin/bash — not a binary blob.

4
Reconnect in the desktop app

Close and reopen the SSH connection in Claude desktop. Should work immediately.

Scenario B — Both Proxmox backups lost

If ccd-cli.working-backup is also gone, recreate the wrapper manually:

bashssh root@<proxmox-server-ip>
cat > /root/.claude/remote/ccd-cli << 'EOF'
#!/bin/bash
export PATH=/root/.nvm/versions/node/v24.14.0/bin:$PATH
ARGS=()
for arg in "$@"; do
    [[ "$arg" == "--dangerously-skip-permissions" ]] && continue
    [[ "$arg" == "--allow-dangerously-skip-permissions" ]] && continue
    ARGS+=("$arg")
done
exec /root/.nvm/versions/node/v24.14.0/bin/node /root/.claude/remote/ccd-cli.js "${ARGS[@]}"
EOF
chmod +x /root/.claude/remote/ccd-cli
chattr +i /root/.claude/remote/ccd-cli

Scenario C — Windows stops working

Unlikely since Hyper-V is permanent, but if it happens (e.g. after a major Windows update):

powershell# Run as Administrator
Enable-WindowsOptionalFeature -Online -FeatureName Microsoft-Hyper-V-All -NoRestart
Enable-WindowsOptionalFeature -Online -FeatureName HypervisorPlatform -NoRestart
# Reboot after

📝 Notes & Caveats

  • Git not installed on Proxmox — causes harmless spawn git ENOENT errors in Claude Code logs. Not related to this issue, safe to ignore.
  • This is a workaround, not a real fix — the root issue is Anthropic shipping a broken ccd-cli binary. A proper patch from Anthropic would make this unnecessary.
  • v2.1.81 does NOT fix the root cause — tested on 2026-03-23 with app 1.1.8308. Plain passthrough (no flag stripping) failed on Mac mini M4. Wrapper still required.
  • Bug reported — github.com/anthropics/claude-code/issues/37967
  • Node version dependency — the wrapper hardcodes /root/.nvm/versions/node/v24.14.0/bin. If Node is updated via nvm, update the wrapper too.
  • Memory context — full fix context is also saved in Claude's memory system on both the Proxmox server and Windows, so future Claude sessions have this context automatically.
